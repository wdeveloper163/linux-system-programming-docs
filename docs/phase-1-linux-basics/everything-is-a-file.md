# "Everything is a file"

`Files` and **Resources** are terms often used differently in web development, but in Linux system programming, they mean **the same thing**.

Instead of thinking:

```text
"Files are only documents on a disk."
```

You should think:

```text
"Files are universal handles for disks, networks, processes, and hardware."
```

That is why understanding "Everything is a file" is so useful in Linux system programming.

---

## Why understand this?

In Linux C++ projects, you often deal with:

*   reading process information
*   debugging running applications
*   accessing hardware devices
*   uniform API (`open`, `read`, `write`)

Understanding this helps because it:

*   simplifies code (one API for everything)
*   enables powerful debugging tools
*   explains how processes communicate
*   makes system introspection easier

::: tip Simple idea
A `Linux System` is like a **Universal API**.

*   **Disk Files** → Standard JSON endpoints.
*   **Network** → Socket endpoints.
*   **Processes** → `/proc` endpoints.
*   **System Calls** → The HTTP Client (`open`, `read`).

You don't need different tools for each. You just `read` the endpoint.
  :::

---

## The basic structure

The difference is made of **abstraction**.

A comparison looks like this:

```text
Real File: Data on Disk
Virtual File: Data from Kernel
```

### What that means

*   **Real File** → Stored permanently (e.g., `config.txt`)
*   **Virtual File** → Generated on demand (e.g., `/proc/status`)
*   **Interface** → Both use `open()`, `read()`, `close()`

Example:

```text
cat document.txt   : Real File
cat /proc/version  : Virtual File
```

This means:

*   `document.txt` reads bytes from hard drive
*   `/proc/version` asks the kernel for current info
*   Your command (`cat`) doesn't care which one it is

::: warning Important
Virtual files often have size 0. You cannot seek them like normal files.
:::

## Your first simple Example

Here is a beginner-friendly example using the terminal.

```bash
# Read a real file
cat ./main.cpp

# Read a virtual process file
cat /proc/self/status
```

### What each line does

```bash
cat ./main.cpp
```

Displays content of a **Real File** on your disk.

```bash
cat /proc/self/status
```

Displays content of a **Virtual File** generated by the kernel.

```text
Name:   cat
State:  S (sleeping)
Pid:    1234
```

Shows current process information dynamically.

## The /proc Filesystem

The `/proc` filesystem makes "Everything is a file" powerful.

Example:

```text
/proc/self : Your Current Process
/proc/[pid] : Specific Process
```

Instead of thinking you need special tools, you reuse the concept of **Paths**.

```text
Virtual Directory Tree
```

::: tip Why this helps
If you can `cat` it, you can `open()` and `read()` it in C++. No special API needed.
:::

---

## The Big Four Paths

These are the most common ones you will use in Linux system programming:

| Path | Meaning | Memory Trick | Example Value |
| :--- | :--- | :--- | :--- |
| `/proc/self/fd` | **Open Files** | **F**ile **D**escriptors | `0`, `1`, `2` |
| `/proc/self/maps` | **Memory Map** | **M**emory **M**ap | `00400000` |
| `/proc/self/status` | **Process Info** | **S**tatus **S**heet | `State: Running` |
| `/proc/self/cmdline` | **Command** | **Cmd** Line | `./app` |

### How the Magic Happens (Step-by-Step)

Let's look at how reading `/proc` works:

```text
1. App opens /proc file
2. Kernel intercepts request
3. Kernel gathers live data
4. Kernel sends text to App
5. App closes file
```

When you read a `/proc` file, the kernel performs a **generation** operation before sending data.

1.  **Identify the Request:** The app calls `read()` on `/proc/self/status`.
    *   So, Kernel knows you want process info.
2.  **Gather Data:** The Kernel looks up your process in memory.
    *   So, it formats the data as text.
3.  **Construct the Response:**

    **Before (Read):**
    ```text
    File: /proc/self/status
    Size: 0 bytes (on disk)
    ```

    **After (Read):**
    ```text
    Content: "State: S\nPid: 1234\n..."
    Size: Generated dynamically
    ```

### Why use `/proc` instead of special APIs?

Imagine you want to know the PID of your program.

**Without /proc (Special API):**
You might need different functions for different info.
```cpp
pid_t pid = getpid();
// Different function for memory...
// Different function for name...
```

**With /proc (Unified API):**
You just read files. Consistent and scriptable.
```bash
cat /proc/self/status | grep Pid
```

This follows the **RESTful Principle** you know from web development (Uniform Interface).

### When to use `/proc` vs Standard APIs

*   **Use `Standard APIs` (e.g., `getpid()`):**
    *   Usually used for **simple, fast** access in code.
    *   Example: `pid_t pid = getpid();`
*   **Use `/proc` (File Reading):**
    *   Used for **detailed inspection** or debugging other processes.
    *   Example: Reading another process's memory maps.

::: tip Beginner takeaway
For now, use standard APIs for your own code. Use `/proc` when you need to inspect details or debug running systems.
:::

## Example with more than one Process

Most Linux systems have many processes with their own `/proc` folders.

Example:

```text
/proc/100/status
/proc/101/status
/proc/102/status
```

### What happens here

*   `100`, `101`, `102` are different Process IDs
*   Each has its own virtual folder
*   `Linux` isolates the data for each process

You can read another process's info (if permissions allow) just by changing the number!

## Common helper System Calls

You will often see file calls like these in C++ for `/proc`.

### `open()`

Opens the virtual file.

```cpp
int fd = open("/proc/self/status", O_RDONLY);
```

### `read()`

Reads the generated text.

```cpp
read(fd, buffer, size);
```

### `close()`

Closes the virtual handle.

```cpp
close(fd);
```

These are usually used together in C++:

```cpp
int fd = open("/proc/self/cmdline", O_RDONLY);
read(fd, buf, sizeof(buf));
close(fd);
```

::: tip What `self` means
It is a symbolic link to your own process ID. `/proc/self` = `/proc/[your_pid]`.
:::

## A practical beginner C++ Example

This is a good starting point for a small Linux C++ project:

```cpp
#include <iostream>
#include <fstream>

int main() {
    // Open /proc file like a normal file
    std::ifstream status("/proc/self/status");
    std::string line;

    // Read line by line
    while (std::getline(status, line)) {
        if (line.find("Name:") == 0) {
            std::cout << line << std::endl;
        }
    }
    return 0;
}
```

### What happens here

*   `#include <fstream>` → Standard C++ file stream
*   `/proc/self/status` → Treated like a normal text file
*   `std::getline` → Reads the kernel-generated text

Compile and run:

```bash
g++ main.cpp -o app
./app
```

## Real-world example

Suppose you are building a system monitoring tool (like `htop`).

Your project has:

*   Need to list all running processes
*   Need to show memory usage

Without understanding `/proc`, you think:

```text
"I need special kernel drivers to see other apps."
```

With "Everything is a file" knowledge, you know:

```text
"I just loop through /proc/[pid]/ and read the files."
```

That is the main benefit: **introspection through standard I/O**.

## Common beginner mistakes

### Assuming File Size

`/proc` files often report size 0. Read until EOF.

### Writing to Read-Only

Many `/proc` files are read-only. Writing causes errors.

### Parsing Text

`/proc` data is text, not binary. You must parse strings.

### Ignoring Permissions

You cannot read `/proc/[other_pid]/` if you lack permissions.

### Confusing `self` and `pid`

`/proc/self` is you. `/proc/[pid]` is specific (maybe others).

## Summary

For Linux system programming with C++, understanding **"Everything is a file"** is the key to simplicity.

Real Files help you describe:

*   persistent data
*   user documents
*   configuration

Virtual Files (`/proc`) help you describe:

*   system state
*   process introspection
*   kernel information

In Linux, if you can `open()` it, you can work with it. `/proc` turns complex kernel data into simple text files you can read.
